# Feature Plan Lite: M1.5.2 — Align config layer and install level language

## Goal

Align ubiquitous language for Config Layers, Active Config Layers, Install Level, Project Level, User Level, and Persisted Scope Values across Rust APIs, CLI help, diagnostics wording in code comments, tests, and docs. Keep `ConfigLayer` as the core type for `shared-project`, `user-project`, and `user` layers.

## Out of scope

- M1.5.3+ (Client Discovery Location, Discovery Requirement, Consumer rename, etc.)
- `--upgrade` → `--refresh-sources` (M1.5.4)
- Renaming `persisted_scope()` method or TOML `scope =` field (keep as Persisted Scope Value in prose only)
- Mass-renaming future milestone "target" / "consumer" technical terms in not-yet-implemented sections unless clearly user-facing today

## Decision: Rust type rename

**Rename `InstallScope` → `InstallLevel`** and `install_scope` fields/helpers → `install_level`, mirroring M1.5.0/1 workflow renames. Rationale: glossary avoids "Install Scope"; type is still early/stub-heavy; mechanical rename now prevents API drift.

**Keep `ConfigLayer` enum name** — glossary uses Config Layer as the term; Rust type already matches.

## Implementation steps

### 1. Core scope types (`crates/agentcfg-core/src/scope.rs`)

- Update module doc: "installation scope" → Install Level vocabulary
- Rename `InstallScope` enum → `InstallLevel` (variants `Project`, `User` unchanged — map to Project Level / User Level in docs)
- Add brief doc comments on `ConfigLayer` and `InstallLevel` referencing glossary terms (Shared Project Config, User Project Config, User Config; Project Level / User Level)
- Keep `persisted_scope()` / `from_persisted_scope()`; document in comment that these are **Persisted Scope Values**, not Install Level

### 2. Core consumers of Install Level

Rename type and field names across:

| File | Changes |
|------|---------|
| `workflow.rs` | `InstallScope` → `InstallLevel`, `install_scope` → `install_level` on requests/results/artifacts |
| `config_paths.rs` | `GeneratedStatePaths` field/method `install_scope` → `install_level`; tests |
| `client_targets.rs` | `SkillTargetRoot.install_scope` → `install_level`; helpers |
| `lib.rs` | re-exports if any |

### 3. CLI (`crates/agentcfg-cli`)

- `args.rs`: `InstallScopeArgs` → `InstallLevelArgs`; update `InitArgs`, `PreviewArgs`, `ApplyArgs`, `InstallLevelArgs` `--user` with `long_help` or `help` per glossary:
  - **init `--user`**: selects **User Config** (User Config Layer)
  - **preview/apply/prune/status `--user`**: selects the user **Install Level** (User Level)
- `commands.rs`: `install_scope()` → `install_level()`, imports, test `maps_install_scoped_commands_to_workflow_request` → `maps_install_level_commands_to_workflow_request`

### 4. Documentation (user-facing / domain)

Update wording in:

| File | Changes |
|------|---------|
| `README.md` | "install scopes" → Install Level / project and user levels |
| `docs/prd.md` | Config layer definitions use Shared Project Config / User Project Config / User Config; Install scope → Install Level; Active layers at Project Level vs User Level; `--user` help text per glossary; remove "configuration scopes" where ambiguous |
| `docs/design-v1.md` | Section "Config Layers and Install Scopes" → "Config Layers and Install Levels"; table column "Install scope" → "Install level" with values `project level` / `user level` or `Project Level` / `User Level`; active layers table uses Active Config Layers wording; line 105: `InstallScope` → `InstallLevel` in code guidance; "install-scoped commands" → "commands at a given Install Level"; keep explicit override semantics **out of V1** sentence but avoid "override" in additive descriptions (already mostly OK) |
| `docs/implementation-plan-v1.md` | Check off M1.5.2 items; update M1.2 bullet references from install scopes → install levels where describing completed work |

**Persisted Scope Value**: In design/prd tables, label the `scope` column as Persisted Scope Value where documenting the TOML field.

**Active Config Layers**: Project Level = Shared Project Config then User Project Config; User Level = User Config only.

### 5. Validation

```sh
cargo test --workspace config_layer install_level
cargo test --workspace
cargo run -p agentcfg-cli -- init --help
cargo run -p agentcfg-cli -- preview --help
```

Confirm no `InstallScope` remains in Rust code (`rg InstallScope`).
