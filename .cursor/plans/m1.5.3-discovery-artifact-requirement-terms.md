# Feature Plan Lite: M1.5.3 — Align discovery, artifact, and requirement terms

## Goal

Align ubiquitous language for Client Discovery Location, Client Discovery Registry, Discovery Requirement, and Installed Artifact across Rust APIs, CLI diagnostics, tests, and domain docs per `CONTEXT.md`. Remove user-facing "client target", "target registry", "Consumer", and "target artifact" wording where glossary terms apply.

## Out of scope

- M1.5.4+ (Skill Source, Discovery Name, `--refresh-sources`, Managed Skill Content rename, etc.)
- Renaming symlink diagnostic terms **Unexpected Symlink Target** and **Broken Symlink** (glossary keeps these scoped to symlink destinations)
- Mass-renaming future M5–M8 implementation-task internals (`target_path`, `target_mode`, planner "target creates") unless the phrase is clearly user-facing today in completed/current sections
- Introducing a full `DiscoveryRequirement` Rust struct (not implemented yet); doc and comment alignment only for requirement vocabulary
- Renaming `alias` / `alias_target` in config schema (skill alias, not client target)

## Decision: Rust renames

**Rename module `client_targets` → `discovery_registry`** and align public/internal types with glossary:

| Old | New | Notes |
|-----|-----|-------|
| `SkillTargetRoot` | `ClientDiscoveryLocation` | Built-in registry entry grouping clients sharing one path |
| `project_skill_target_roots` | `project_client_discovery_locations` | |
| `user_skill_target_roots` | `user_client_discovery_locations` | |
| `ExistingTargetArtifact` | `UnmanagedInstalledArtifact` | Init finds artifacts not in Manifest |
| `TargetReadFailure` | `DiscoveryLocationReadFailure` | Scan failure at a location |
| `InitWarning::ExistingTargetArtifact` | `InitWarning::UnmanagedInstalledArtifact` | |
| `InitWarning::TargetReadFailure` | `InitWarning::DiscoveryLocationReadFailure` | |
| `SkillTargetInspection` | `DiscoveryLocationInspection` | internal |
| `inspect_existing_skill_targets` | `inspect_existing_discovery_locations` | |
| `scan_target_root` | `scan_discovery_location` | internal |

**Keep `path: PathBuf`** on location structs — low-level filesystem path is fine; avoid "target" in field names exposed to CLI.

Rationale: early codebase; mechanical rename prevents API drift; validation filters expect `discovery_registry` and `discovery_requirement` in test names.

## Implementation steps

### 1. Discovery registry module (`crates/agentcfg-core/src/discovery_registry.rs`)

- Rename file from `client_targets.rs`; update `lib.rs` mod declaration
- Apply type/function renames above
- Module doc: describe **Client Discovery Registry** catalog and **Client Discovery Locations**
- Add `#[cfg(test)]` module or tests named `discovery_registry_*` (e.g. `discovery_registry_groups_shared_agents_path`) if none exist — at minimum ensure `cargo test discovery_registry` matches renamed code paths via workflow tests

### 2. Workflow (`crates/agentcfg-core/src/workflow.rs`)

- Update imports from `discovery_registry`
- Rename artifact/warning types and helpers per table
- Rename tests: `unmanaged_project_artifacts_*` → `unmanaged_project_installed_artifacts_*` or include `discovery_registry` in name; `missing_target_roots_*` → `missing_discovery_location_roots_*`; `target_scan_*` → `discovery_location_scan_*`
- Add at least one test name containing `discovery_requirement` documenting glossary intent (e.g. comment that future manifest entries use Discovery Requirements keyed by config layer, client, install level) — can be a small unit test on a doc constant or placeholder struct in `discovery_registry.rs` if no requirement model exists yet

### 3. CLI (`crates/agentcfg-cli`)

- `commands.rs`: `render_skill_target_warning` → `render_discovery_warning`; user-facing strings:
  - "could not scan client target" → "could not scan client discovery location"
  - "unmanaged skill artifact" → "unmanaged installed artifact" (match glossary Installed Artifact / Unmanaged Artifact)
- `args.rs`: prune `about` if it says only "artifacts" without context — optional tweak to "stale managed installed artifacts"
- `tests/init.rs`: update stderr assertions for new warning text

### 4. Documentation (user-facing / domain)

Update glossary-aligned terms in:

| File | Changes |
|------|---------|
| `docs/prd.md` | Terms section: Target → Client Discovery Location; Consumer → Discovery Requirement; target artifact → Installed Artifact; client targets → Client Discovery Locations; managed source tree phrasing can stay until M1.5.4; workflow bullets use Discovery Requirements, Stale Discovery Requirement, etc. where referring to manifest/planner behavior |
| `docs/design-v1.md` | "Client Target Registry" → "Client Discovery Registry"; client target / target artifact / consumer prose → glossary terms; line 105 code guidance: reserve low-level `target_path` / symlink target only for filesystem symlink diagnostics and internal manifest fields, not user-facing "client target"; update manifest `consumers` JSON example comment to Discovery Requirements keyed by config layer, client, install level; keep Unexpected Symlink Target / broken symlink sections |
| `docs/implementation-plan-v1.md` | Check off M1.5.3 items; update M1.2 completed bullets that say "client target" if user-facing |
| `README.md` | "target installation" → discovery/install wording; client target symlink → Client Discovery Location; manifest-owned artifacts → Installed Artifacts where user-facing |

**Discovery Requirement**: Replace `{scope, client}` consumer descriptions with Discovery Requirements keyed by **Config Layer**, **Client**, and **Install Level** (Persisted Scope Value may appear in serialized forms but prose should say Config Layer).
